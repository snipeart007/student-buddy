We have to build a app which is a python based app.
This app acts as a FastAPI backend.
This backend itself is installed on every user's computer.
The backend also serves a frontend. The frontend should be created in frontend folder. This frontend is served using the FastAPI server.
The frontend should be a NextJS app.
The local backend uses a local LLM model through llama.cpp. This model is a custom fine-tuned model.
The agent_prompts folder contains the agent prompts and their input output schemas.
The app should first ask a questionnaire about the student and fill in the StudentState.
This state can be found in the schema. Then the model will act as a chat interface.
The chat interface will be linked to the local backend using the gradio/client module.
The backend will expose a headless gradio.Blocks API (No UI, only API).
The local backend will also manage the user state.
The state which is to be persisted across sessions, i.e. the student state will be stored as json on the local machine. Encrypt this state using some basic AES.
The app when launched, will start the FastAPI+Gradio backend. The gradio app should be mounted onto our main FastAPI app. 
The working: First run: Login/Register the user and fill in the questionnaire. Then backend will setup the student state and start the chat interface.
The chat interface will act in different way. Firstly, the frontend will send in the requests and the backend will respond, using streaming. The streaming will show the thinking blocks as well as the final response.
The request will be multi-modal, i.e. it can contain text, images, and other media types.
The custom tuned LLM model is a custom fine-tuned Gemma 4 E2B model.
The login, registration, etc workflow:
The backend will communicate with institution API to handle the login, registration, etc workflow.
Once registered and logged in, the user will be able to run the app without using the institution API.
The backend will generate the responses using the local model.

Actual working of chats:
The system is a chat interface that allows students to manage their academic load with their mental health.
The backend will first receive the request from the frontend, then it will use the local model to generate the policy model response.
The policy model will generate a response like this:
{
 "mental_weight": 0.72,
 "academic_weight": 0.28,
 "mode": "emotional_recovery",
 "risk_level": "high"
}
The backend will then use this response to generate the final response using the local model, if the risk level is not high or critical.
The final response in this case will be generated by passing the policy model response, user query, user's StudentState to the local model.
If the risk level is high or critical, the two seperate responses(Academic and Mental) will be generated by passing the user query, user's StudentState to a institution API model. (for development fallback to the local model. Add #TODO wherever the institution API is to be called during developement phases)

These two responses will be combined to generate the final response, using the fusion model(again called using the institution API model).

The backend will be streaming this whole process to the frontend.(the thinking process will also be streamed)
The backend will update the student state after each response.
The final Fusion model must also output the updated student state.

If the risk level is not high or critical, after outputting the final response, the backend will update the student state using the local model. This process will be done using a seperate process that is not part of the main streaming loop.

The frontend will be made using Next.js and MUI. The UI will be responsive and user-friendly.

The frontend will include a onboarding setup while registering.

The frontend's chat interface will be implemented using MUI special chat components. The chat interface will be managed using the gradio/client library.

The backend will update the student state to the institution API after every response.
