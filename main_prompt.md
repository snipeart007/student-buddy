# AI-Powered Academic Advisor Application

## Overview

We have to build an app which is a Python-based application.

This app acts as a FastAPI backend.

This backend itself is installed on every user's computer.

The backend also serves a frontend. The frontend should be created in the `frontend` folder. This frontend is served using the FastAPI server.

The frontend should be a Next.js app.

The local backend uses a local LLM model through `llama.cpp`. This model is a custom fine-tuned model.

The `agent_prompts` folder contains the agent prompts and their input/output schemas.

---

# Core Functionality

The app should first ask a questionnaire about the student and fill in the `StudentState`.

This state can be found in the schema.

Then the model will act as a chat interface.

The chat interface will be linked to the local backend using the `gradio/client` module.

The backend will expose a headless `gradio.Blocks` API.

- No UI
- API only

The local backend will also manage the user state.

The state which is to be persisted across sessions, i.e. the student state, will be stored as JSON on the local machine.

Encrypt this state using some basic AES encryption.

# Request Flow

## Step 1: Frontend Request

The frontend sends the request to the backend.

The backend responds using streaming.

The streaming must include:

- thinking blocks
- intermediate reasoning
- final response

---

## Step 2: Policy Model Generation

The backend first uses the local model to generate the policy model response.

Example policy response:

```json
{
  "mental_weight": 0.72,
  "academic_weight": 0.28,
  "mode": "emotional_recovery",
  "risk_level": "high"
}
```

---

# Policy Routing Logic

## Case 1: Risk Level is NOT High or Critical

If the risk level is not `high` or `critical`:

The backend will generate the final response using the local model.

Inputs passed into the local model:

- policy model response
- user query
- user's `StudentState`

The backend will then stream the generated response to the frontend.

---

## Case 2: Risk Level is High or Critical

If the risk level is `high` or `critical`:

Two separate responses must be generated:

1. Academic response
2. Mental health response

These responses will be generated by passing:

- user query
- user's `StudentState`

to an institution API model.

During development:

- fallback to the local model
- add `#TODO` wherever the institution API is supposed to be called

---

# Fusion Model Workflow

The two separate responses:

- Academic response
- Mental health response

will be combined using the Fusion model.

The Fusion model is also called using the institution API model.

During development:

- use local model fallback
- add `#TODO` comments for institution API integration points

---

# Streaming Requirements

The backend must stream the entire orchestration process to the frontend.

This includes:

- thinking process
- intermediate reasoning
- policy routing output
- advisor outputs
- fusion response
- final response

---

# Student State Management

The backend must update the `StudentState` after every response.

The final Fusion model must also output the updated `StudentState`.

---

# Background State Update Process

If the risk level is NOT `high` or `critical`:

After outputting the final response:

- the backend will update the student state using the local model

This process must:

- run in a separate process/thread/task
- not be part of the main streaming loop
